# Examples

## Scenario A: Coding feature with specs + progress docs

### Good
- Reads `docs/prd.md`, `docs/specs/*`, `docs/progress.md`.
- Selects one small slice.
- Outputs mini-PRD context + ordered implementation steps + validation checks.
- Provides 3 execution todos with clear dependencies and a yes/no handoff.

### Bad
- Restates feature request.
- Mentions problems but no ordered steps.
- No validation plan.

---

## Scenario B: Non-coding operational task

### Good
- Reads goals/projects/history/task context.
- Identifies immediate operational slice (for example, backlog cleanup + canonical ticketing).
- Outputs concrete actions with owner/system boundaries and risks.
- Ends with "I already prepared this; approve yes/no."

### Bad
- Generic suggestions ("clean up tasks", "improve process").
- No concrete actions or sequence.

---

## Scenario C: Ambiguous request

### Good
- States assumptions explicitly.
- Asks 1-2 critical clarifying questions only if required.
- Still provides a provisional plan for what can proceed now.

### Bad
- Blocks planning entirely waiting for perfect info.
- Asks many low-value questions.
